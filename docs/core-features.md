# OpenManager 核心功能清单

这份文档不是 roadmap，也不是实现细节笔记。

它定义的是 OpenManager 当前应该稳定成立的核心功能，用来作为后续重构、补测试、补文档、做国际化时的共同基线。

一句话定位：

> OpenManager 是一个面向 OpenClaw 的本地优先项目工作台，用项目承载长期上下文，用会话承载工作线程，用 agent 承载执行边界。

## 使用方式

- 当你要继续重构时，先看这里，确认改动触碰的是哪一块核心能力
- 当你要补测试时，优先补这里还没有被自动化覆盖的真实链路
- 当你要加新能力时，先判断它属于扩展，还是会影响这里定义的稳定行为
- 当你要做国际化或英文版时，以这里的功能定义为准，不要顺手改产品逻辑

## 产品级约束

- 不改主界面布局结构
- 不改主要交互习惯
- 不删现有功能入口
- 不把“内部实现优化”顺手变成“产品重新设计”
- 允许重构前后端、拆模块、补类型、补测试、补文档、修真实 bug

## 核心对象模型

### 1. Project 是工作空间边界

- 一个项目对应一组长期工作的上下文
- 项目拥有自己的共享记忆、会话、任务、上传文件和配置
- 切换项目时，页面展示和操作范围都应切换到该项目上下文

### 2. Session 是对话边界

- 一个项目下可以有多个会话
- 每个会话对应一条持续工作的对话线程
- 会话之间的聊天记录、热记忆、事实层和摘要层应保持边界清晰

### 3. Agent 是执行边界

- 项目可以绑定独立 agent，也可以使用默认执行环境
- 运行时应尽量在绑定 agent / workspace 的边界内执行
- agent 绑定属于项目配置，不属于单条消息的临时状态

## 核心功能

### 1. 项目管理

- 创建项目
- 切换项目
- 重命名项目
- 删除项目
- 查看项目概览信息
- 打开项目设置
- 保存项目设置

稳定行为：

- 项目切换后，会话、任务、记忆、文件、技能/cron 相关展示都应切到对应项目
- 项目删除后，不应残留一个仍可操作的假当前项目状态
- 项目设置的保存结果应立刻体现在当前项目配置上

### 2. 会话管理

- 创建会话
- 切换会话
- 重命名会话
- 删除会话
- 从会话列表打开对话

稳定行为：

- 当前激活会话应持久化到项目配置
- 打开旧会话时，应能正确加载该会话历史内容和相关状态
- 删除当前会话后，界面状态要保持可理解，不出现悬空引用

### 3. 项目级共享记忆

- 查看共享记忆
- 编辑共享记忆
- 保存共享记忆

稳定行为：

- 共享记忆属于项目，不属于某个单独会话
- 它应适合放目标、约束、规则、架构说明、长期约定
- 编辑共享记忆不应污染某个会话的原始消息历史

### 4. 会话级分层记忆

- 保留原始消息流
- 维护 hot / facts / summary 等分层状态
- 在前端查看会话 memory 相关内容
- 支持 memory 压缩或整理入口

稳定行为：

- 写入尽量全，注入尽量省
- 分层记忆服务于“继续工作”而不是“无限堆聊天记录”
- 身份确认、在场确认、说话风格修正等低价值 meta 对话不应污染高价值记忆层

### 5. 对话与运行时

- 发送文本消息
- 打开并继续已有对话
- 正确渲染历史消息和新增消息
- 展示运行中的状态卡片
- 支持 stop run
- 支持 reconnect run
- 支持不同 runtime 的执行配置

稳定行为：

- 对话的主要工作流必须保持顺手，不应因为重构改变使用节奏
- 运行状态应清晰反映当前阶段，而不是只显示不透明的“处理中”
- 活跃 run 的控制按钮只在应该出现时出现
- 已完成 run 不应残留误导性的活跃状态

### 6. 输入器与发送习惯

- `Shift+Enter` 插入换行
- 双击 `Enter` 发送
- 单击 `Enter` 不误发
- 发送按钮状态跟随文本 / 文件状态变化
- 保持现有输入与发送习惯

稳定行为：

- 输入器行为是高敏感区，任何改动都应视为产品行为变更风险
- 不要为了“更现代”或“更自然”就擅自修改发送规则

### 7. 文件上传与预览

- 上传图片
- 上传非图片文件
- 在发送前预览已选文件
- 在发送前移除已选文件
- 查看项目文件列表
- 删除项目中的文件

稳定行为：

- 图片和普通文件都应能被用户理解地展示
- 文件预览属于发送前的重要确认环节，不能在重构中丢失
- 文件应明确归属到当前项目

### 8. 任务管理

- 新增任务
- 标记完成
- 取消完成状态
- 删除任务
- 在项目内查看任务列表

稳定行为：

- 任务属于项目，不属于单个会话
- 未完成任务应能服务于聊天上下文和工作推进
- 任务操作应是轻量、快速、低摩擦的

### 9. 项目设置与执行配置

- 设置默认模型
- 设置 runtime
- 选择或绑定 agent
- 设置 workspacePath
- 选择使用主工作区、项目工作区或自定义工作区
- 一键创建并绑定 agent

稳定行为：

- 配置项应反映项目级执行策略，而不是一次性 UI 状态
- 自动创建并绑定 agent 应生成可用且稳定的配置结果
- runtime defaults / model options / agent options 的加载失败应尽量可降级

### 10. Skills 面板

- 查看技能列表
- 查看技能状态
- 启用技能
- 禁用技能

稳定行为：

- Skills 面板主要承担查看和切换能力状态的职责
- 技能切换结果应写回 OpenClaw 配置，并对后续会话生效
- 在能力不可用时，界面应尽量表现为“可理解的不可用”，而不是静默失效

### 11. Cron 面板

- 查看 cron 列表
- 创建 cron
- 手动运行 cron
- 启用 cron
- 禁用 cron
- 删除 cron

稳定行为：

- Cron 是 OpenClaw 工作流的延伸，而不是单独的复杂子产品
- 常见管理操作应能在当前面板内完成
- 创建与状态切换失败时，应能明确反馈结果

### 12. 文件面板 / 日志面板 / Memory 面板

- 打开文件面板
- 打开日志面板
- 打开 memory 面板
- 在相关面板中查看和操作对应内容

稳定行为：

- 这些面板是主对话的辅助工作区，不应破坏主流程
- 关闭再打开后，状态不应明显错乱

### 13. 手册与应用内帮助

- 根页面可正常加载
- `manual.html` 可正常加载
- 主界面的 Manual 入口可正常打开手册

稳定行为：

- 手册页属于产品的一部分，不是可有可无的外链
- 功能演进后，手册至少要保持基本可用和不误导

## 非功能性但属于核心要求的能力

### 1. 本地优先

- 项目数据应存放在本地工作区
- 仓库内不应混入用户运行期数据
- 项目文件结构应便于备份、恢复和排查

### 2. 行为保真重构

- 可以重写实现，不要重写产品
- 可以拆模块，不要顺手重排交互
- 可以补类型和测试，不要把“更顺眼”当作合法理由去改行为

### 3. 可维护性

- 前端逻辑应继续从大模块向更清晰边界演进
- API 调用应尽量集中和可替换
- 后端路由职责要尽量稳定、单一、可定位

### 4. 可验证性

- 高风险行为要能通过 `npm test` 和浏览器级回归快速验证
- 新补测试时，优先补真实链路，而不是只补桩
- 文档中的功能定义、回归清单和自动化覆盖应尽量一致

## 当前最值得继续补强的方向

### 1. 真实链路回归覆盖

- 真正跑通 gateway chat
- 真正验证本地 runtime 链路
- 补 memory 面板、日志面板、文件面板的关键回归

### 2. 功能边界文档化

- 把哪些行为算“稳定契约”继续写清楚
- 把哪些只是当前实现细节和可重构区域继续分开

### 3. 国际化准备

- 文案提取和语言切换建立在现有稳定行为之上
- 不把 i18n 工作和交互重写绑在一起

## 建议你后续继续补充的栏目

- 每个核心功能的“用户价值”
- 每个核心功能的“当前已覆盖自动化”
- 每个核心功能的“未覆盖高风险点”
- 每个核心功能后续可拆出的 refactor / test / docs / i18n 子任务

如果后面要按这份文档继续 vibe coding，最自然的方式是：

1. 先确认某个改动影响哪一项核心功能
2. 再确认它触碰的是“稳定行为”还是“内部实现”
3. 然后只拆一小块做完，跑 `npm test`
4. 最后把验证结果回写到对应功能项下
