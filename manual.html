OpenManager Handbook

本地项目工作台
使用手册

OpenManager 是为 OpenClaw 设计的本地项目工作台。它把“项目、会话、记忆、任务、文件、agent 绑定”放进同一个持续使用的容器里，重点解决两件事：多线并行时别串线，和长期工作时别忘事。

最适合谁

同时推进多个项目、需要长期上下文、又不想把所有内容堆进一条聊天记录的人。

运行方式

前端静态页面 + 本地 Node API + OpenClaw CLI / Gateway。

核心价值

多线并行更清楚，记忆管理更稳定，项目上下文更容易持续积累。

1. 这是什么

OpenManager 不是一个泛泛的“项目列表”，而是一个面向 OpenClaw 的本地项目工作台。你可以把它理解成：一个项目就是一个长期上下文容器，一个会话就是项目里的单条工作线程，一个 agent 则是这个项目的执行边界。

项目 = 长期工作容器

放项目目标、共享记忆、任务、文件、模型设置和 agent 绑定。

会话 = 当前工作线程

把“需求梳理”“接口联调”“发布收尾”拆成不同线程，避免信息互相污染。

记忆 = 持续上下文

既保留完整原始记录，也尽量把当前目标、关键事实和摘要提炼出来。

Agent = 执行边界

正式项目可以绑定自己的 agent、workspace 和模型，不用和其他项目抢上下文。

2. 适合什么场景

这套产品特别适合下面这些情况：

  • 你同时维护多个项目，不想把 A 项目的上下文带进 B 项目。
  • 一个项目会持续很多天甚至很多周，需要长期保存目标、约束、任务和关键结论。
  • 同一个项目里又有多个阶段，想把“规划、执行、排查、收尾”分成不同会话。
  • 你希望项目相关的文件、上传资料、对话和记忆都落在同一处，而不是分散在多个工具里。
一句话判断 如果你经常需要重新解释背景、找回上次做到哪里、或者担心多个项目串线，OpenManager 就很适合你。

3. 快速开始

第一次使用时，推荐按下面的顺序来，而不是一上来就堆很多对话：

步骤 1：创建项目

项目名建议写成长期主题，例如“官网改版”“数据看板”“内容选题库”。项目描述写成一句话目标，方便后面快速识别。

步骤 2：补共享记忆

先写项目目标、成功标准、当前范围、关键背景和下一步。这样后面每个会话都能继承这些长期信息。

步骤 3：新建会话

会话名建议写成当前阶段，例如“需求梳理”“接口联调”“问题排查”“发布收尾”。

步骤 4：开始工作

把任务拆到任务列表，把参考资料传到上传区，把当前阶段的对话都放进对应会话里持续推进。

轻量项目怎么用

直接用默认 main，不单独绑定 agent，适合短期整理和轻任务。

正式项目怎么用

给项目绑定自己的 agent、workspace 和模型，让每个项目成为独立工作环境。

4. 多线并行怎么用

OpenManager 最重要的价值之一，就是把“多个项目并行”和“一个项目内多个线程并行”都整理清楚。推荐的使用节奏是：

推荐工作流

一个长期项目 = 一个 Project 例如「官网改版」和「数据看板」不要混在同一个项目里。
一个阶段 = 一个 Session 把规划、执行、排查、收尾拆成不同会话，方便专注和回看。
一个正式项目 = 一个 Agent 需要独立 workspace 和持续执行能力时，就绑定自己的 agent。
多个窗口并行 可以同时开多个项目窗口，让不同工作流保持隔离，不必来回切上下文。
推荐

项目按长期主题拆分；会话按当前阶段拆分；任务按可执行动作拆分。

不推荐

把所有工作都堆到一个项目、一个会话、一个长对话里，再靠搜索回忆过去发生了什么。

5. 记忆系统怎么用

记忆系统不是为了“存更多”，而是为了“少重复、少丢信息、少串线”。对于用户来说，你只需要记住下面这几个层次分别负责什么：

projects/<project>/ ├── .project.json ├── memory/ │ ├── shared.md │ ├── session-xxx.meta.json │ ├── session-xxx.jsonl │ ├── session-xxx.hot.json │ ├── session-xxx.facts.json │ └── session-xxx.summary.md └── tasks/tasks.json
  • `shared.md`：项目共享背景。适合放目标、规则、约束、长期约定。
  • `session-xxx.jsonl`：原始对话流。完整保留，方便追查真实上下文。
  • `session-xxx.hot.json`：当前工作记忆。让系统快速知道你现在做到哪了。
  • `session-xxx.facts.json`：关键事实层。把决策、约束、偏好、待办提出来。
  • `session-xxx.summary.md`：阶段摘要。帮助跨轮次快速接续，而不是每次重读全历史。
  • `tasks.json`：项目任务层。适合放真正要执行的动作，而不是把 todo 埋在长对话里。

Memory 分层结构图

shared.md 项目长期背景：目标、规则、上下文、长期约定
↓
session.jsonl 原始事件流：用户、助手、错误、工具和阶段摘要事件
↓
session.hot.json + session.facts.json 当前工作记忆 + 关键事实：服务快速续接和避免忘事
↓
session.summary.md 阶段摘要：服务跨轮次压缩、人工回看和快速交接

每轮对话怎么取记忆

1. 读项目背景 先拿 `shared.md`，保证项目目标和长期规则不丢。
2. 读当前工作状态 优先读 `hot/facts`，快速知道当前目标、结论、阻塞。
3. 读会话摘要 用 `summary` 接上跨阶段上下文，而不是重扫整段历史。
4. 尾读 recent 只从 `jsonl` 末尾取最近几条，补足对话连贯性。
怎么用最有效 共享记忆写长期信息，会话写当前阶段，任务写可执行动作。这样系统才知道哪些信息该长期带着，哪些只属于当前线程。

下面这两张图最值得先看：第一张帮你理解项目级记忆该写什么，第二张帮你理解长对话为什么不会越聊越乱。

共享记忆页

适合在项目刚建立、阶段切换或需求发生明显变化时维护。这里建议只放长期有效的信息，例如目标、成功标准、边界、关键背景和下一步。

不要把临时讨论、零散待办和即时聊天记录堆进共享记忆。共享记忆越稳定，后面每个会话续接得越自然。
对话窗口与会话记忆

对话窗口右侧会展示当前工作记忆、最近结论和下一步建议。这样即使一个会话已经持续了很久，系统也能先抓住重点，再去补最近对话。

这块区域最适合用来判断“现在做到哪了”和“接下来该推进什么”，而不是回头翻整段聊天记录。
AI 压缩什么时候用

推荐在一个阶段已经形成稳定结论时再压缩，比如“需求梳理结束”“接口联调告一段落”“排查结论已经明确”。这样生成的摘要更稳，也更适合下次接续。

OpenManager 会追加阶段摘要，但不会覆盖原始记录。也就是说，压缩的作用是帮你提炼，不是替你删历史。
对话管理小技巧

如果一个项目已经从“规划”进入“执行”或“排查”，不要把所有内容都接在同一个会话里。新建一个会话继续推进，再让共享记忆和阶段摘要帮助你接续上下文，会更清楚。

一个长期项目通常会有多条会话线程，这正是 OpenManager 避免串线的关键设计。

从用户视角看，OpenManager 是“一个前端界面 + 一个本地 API + 一套本地项目文件 + OpenClaw 执行能力”的组合。下面几张图可以帮助你快速理解它的结构。

7. 对话与文件流转

这里是 OpenManager 最关键的两条链路：项目对话怎么拉 OpenClaw，文件又是怎么进到项目里的。

项目对话 → OpenClaw 流程图

1. 用户发送消息 消息从前端会话窗口发出，并带上项目和会话信息。
2. 后端组装上下文 拼接共享记忆、工作记忆、关键事实、会话摘要、最近对话和任务摘要。
3. 调用 OpenClaw 根据项目配置走 gateway agent 或本地 codex 执行。
4. 写回会话记忆 结果、阶段状态和后续摘要会回写到 session memory。
5. 前端更新界面 用户看到最新回复，并能继续沿同一项目线程往下做。

文件上传流程图

1. 前端选择文件 支持把图片或文件上传到当前项目。
2. 后端写入 uploads/ 文件会按项目落到 `uploads/` 目录，并生成可访问路径。
3. 文件浏览器展示 前端可以按项目查看上传内容，图片会直接预览。
4. 结合会话继续工作 用户可以围绕这些文件继续聊天、整理任务和推进项目。
为什么这样设计 对话、文件、任务和记忆都按项目归档，最大的好处就是：后面回看、继续做、换人接手时，都更容易找到上下文。

8. Cron、Skills 与后续方向

Cron 和 Skills 是 OpenManager 里更偏“稳定工作流”的两组能力。它们不是第一次打开就必须配置的内容，但一旦进入正式使用阶段，会很值得投入。

  • Cron 可以为项目安排定时工作，例如日报、巡检或提醒。
  • Skills 可以控制当前用户在 OpenClaw 中启用哪些能力。
  • 修改 Skills 状态后，最稳妥的方式是新建会话再继续工作。
Skills 适合什么时候看

正式项目开始前，先确认当前账号有哪些技能可用、哪些缺依赖、哪些暂时关闭，避免工作到一半才发现能力没开。

Cron 适合什么时候配

当一个项目已经形成固定节奏，比如每天同步、每周回顾、例行巡检或发布收尾提醒时，再把它们做成重复任务最合适。

Skills 面板

这里会把技能分成可用、缺依赖、已禁用等状态，适合快速判断当前用户能调用哪些能力。常用技能保持启用，不常用技能先关闭，会更利于正式工作流保持稳定。

如果你刚改完 Skills 状态，建议新建一个会话再继续做事，让新的能力边界从下一轮开始生效。
Cron 面板

适合把“日报、巡检、每周回顾、收尾提醒”这类固定动作做成重复任务。建议先从 1 到 2 个最稳定的场景开始，等节奏跑顺后再逐步增加。

Cron 的价值不在“自动化越多越好”，而在于把那些每次都会做、又容易忘的动作固定下来。

后续如果继续演进，这个产品会围绕更顺滑的上手体验、更清楚的上下文组织和更稳定的项目执行体验继续打磨，但整体方向不会变：让多项目并行和长期记忆管理变得更自然。
