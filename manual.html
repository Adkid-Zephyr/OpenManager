OpenManager Handbook

本地项目工作台
使用与设计手册

OpenManager 的目标不是做一个华而不实的前端壳，而是把“项目、会话、共享记忆、任务、agent 绑定、本地文件系统”这些长期工作流真正揉成一个可持续使用的工作台。当前版本已经按“前后端分离、数据本地存储、后续可开源”这个方向整理过一轮。

当前定位

先作为 GitHub 开源项目维护，再收敛为一个 OpenClaw skill。

运行方式

前端静态页面 + 本地 Node API + OpenClaw CLI / Gateway。

目录

1. 设计目标 2. 结构组成 3. 数据与记忆 4. Agent 绑定方式 5. 前端主要功能 6. Cron 与 Skills 7. 开发与开源建议

1. 设计目标

核心思路是“每个项目一个上下文容器，每个对话一个工作线程”。项目承载共享记忆、任务、文件和 agent 绑定；会话承载单条线程的对话与局部记忆。这样既保留长期上下文，也避免不同对话互相污染。

保留原始前端体验

不推翻你已经习惯的界面和交互，只在内部结构上做模块化和解耦。

适合长期维护

源码、运行时数据、OpenClaw 配置和项目数据彼此分层，后面开源时更容易整理。

2. 结构组成

当前项目已经分成静态前端和 API 后端两层。前端负责界面和交互，后端负责本地文件、项目数据、Cron、Skills 和 OpenClaw CLI 调用。

skills/project-workspace/ ├── app.html ├── api.js ├── manual.html ├── frontend/ │ ├── index.html │ ├── api.js │ └── manual.html └── backend/ ├── server.js ├── routes/ └── lib/

3. 数据与记忆

项目数据仍然放在本地 `projects/` 下。每个项目使用文件夹作为边界，这和你一开始“本地文件系统是后端”的设计初衷一致。

projects/<project>/ ├── .project.json ├── memory/ │ ├── shared.md │ ├── session-xxx.meta.json │ ├── session-xxx.jsonl │ └── session-xxx.summary.md └── tasks/tasks.json
  • `shared.md`：项目级共享记忆，适合放目标、约束、架构决策。
  • `session-xxx.jsonl`：原始消息流，便于程序追加写入。
  • `session-xxx.summary.md`：对话摘要层，避免每轮都塞完整历史。
  • `tasks.json`：任务共享层，不再默认全量注入每一轮消息。

4. Agent 绑定方式

推荐结构是“每个项目一个 agent，每个对话一个 session”。这样 workspace、模型和能力边界更清晰，也更接近 Dashboard 的真实使用方式。

  • 项目设置中可以手动绑定已有 agent。
  • 创建项目时可以直接自动创建并绑定 agent。
  • 绑定信息会写回 `.project.json`，包括 `agentId`、`workspacePath`、`model`。
{ "agentId": "dashboard", "workspacePath": "/Users/xxx/.openclaw/workspace/projects/Dashboard 数据分析项目", "model": "qwen3.5-plus" }

5. 前端主要功能

  • 项目列表、项目详情、项目设置、新建项目。
  • 多会话管理。多窗口方式替代旧的多标签方案。
  • 共享记忆查看与编辑。
  • 任务列表与完成状态管理。
  • 会话对话窗口、附件上传、记忆压缩、文件浏览。

最近这轮还优化了“项目设置”的打开速度，并把新建项目时的 agent 绑定 UI 重新排了版。

6. Cron 与 Skills

Cron 现在走 OpenClaw 原生命令，支持最基础的创建、启用、禁用、立即运行和删除。Skills 的列表和状态来自 `openclaw skills list --json`，启用/禁用会写入 `~/.openclaw/openclaw.json` 的 `skills.entries`。

{ "skills": { "entries": { "project-workspace": { "enabled": true }, "sag": { "enabled": false } } } }

注意：OpenClaw 会对技能列表做会话级快照，所以改完技能状态后，新建会话最稳妥。

7. 开发与开源建议

  • 把运行时数据和源码严格分开，避免开源仓库混入用户真实项目数据。
  • 继续保持 API 路由按领域拆分，减少单文件膨胀。
  • 前端可以继续沿着 `api.js + 单页静态 HTML` 方式演进，先保证可维护性，再考虑框架化。
  • 如果后续发布到 GitHub，建议补齐 `LICENSE`、`.gitignore`、`CONTRIBUTING.md` 和最小安装说明。
